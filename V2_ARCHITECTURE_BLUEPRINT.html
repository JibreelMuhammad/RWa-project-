Executive Summary

1.1 What This Platform Does

The Property Web3 Portal is a multi-tenant SaaS platform that enables real estate companies, investment funds, and property developers to tokenize real-world property assets on the XRP Ledger (XRPL). The platform bridges traditional real estate investment with blockchain-native ownership, allowing fractional property ownership through Multi-Purpose Tokens (MPT) under the XLS-33 and XLS-89 standards.

Real estate operators — the tenants of this SaaS platform — run branded investor portals through which their end users can browse properties, pass KYC/AML verification, purchase fractional token shares, track their portfolio, and participate in secondary market trading. All blockchain operations are proxied through secure server-side edge functions; no private keys or signing operations ever occur in the browser.

ℹ
Platform Hierarchy

Platform (us) → operates the SaaS infrastructure and charges tenants commission / subscription fees.
Tenants → real estate companies or investment managers who run white-labeled property portals.
End Users → retail or accredited investors who buy fractional token shares through tenant portals.

1.2 Why We Are Rebuilding

Version 1 was built as a rapid prototype using Lovable, an AI-assisted development platform that auto-generates and commits code. While this accelerated initial delivery to over 250 commits, the resulting codebase carries significant technical debt that makes it unsuitable for production use.

The V1 code audit identified 3 critical issues, 8 high-priority issues, and 9 medium-priority issues. Most critically, a wallet secret storage vulnerability (issue C1) stores private key material in plaintext in the application database — an unacceptable security posture for any production financial platform. Combined with the absence of multi-tenancy, missing type safety on blockchain responses, no rate limiting, and a god-object context architecture, a clean rebuild is the more economical path compared to incremental remediation.

⚠
Critical: Do Not Deploy V1 to Production

V1 stores wallet_secret (raw XRPL private keys) in an unencrypted database column. V1 has no multi-tenant data isolation — all user data is shared in a single schema with minimal RLS. V1 has no rate limiting on any edge functions. V1 must not be used for any real-money (mainnet) transactions beyond internal testing.

What We Learned From V1 — Lessons Learned

The V1 codebase was audited thoroughly before beginning V2 design. The following lessons represent the most important architectural and operational insights extracted from that analysis. Each lesson directly informs a V2 design decision.

2.1 Critical: Raw Private Key Storage

⚠
Audit Issue C1 — wallet_secret stored unencrypted in user_wallets table

The testnet faucet flow stored users' raw XRPL wallet secrets (private keys) directly in the user_wallets table as a plaintext column. The edge function xrpl-submit-signed fetched this column to perform server-side transaction signing. While implemented as testnet-only, no runtime guard prevented a mainnet key from being stored the same way. A single database breach would expose all private keys.

Lesson: Never architect a system where a database SELECT can return a signing key. Private key material must never enter a relational database, even encrypted. V2 uses a vault reference pattern: the database stores only an opaque vault key ID, while the actual key material lives in a dedicated secrets service (Supabase Vault or HSM). For testnet, a single platform-owned funded wallet handles all auto-signing, eliminating per-user key storage entirely.

2.2 God Object Contexts

ActiveWalletContext.tsx grew to over 300 lines and accumulated seven distinct responsibilities: DB wallet loading, active wallet selection, localStorage persistence, wallet CRUD operations, audit logging, connect modal state, and inactivity timeout orchestration. This violated the Single Responsibility Principle and made the file extremely fragile — any change to one area risked breaking another.

Lesson: Global state management must be decomposed early. In V2, wallet state is split across a Zustand store (useWalletStore), a connect hook (useWalletConnect), and a standalone inactivity hook (useInactivity). No single file owns more than one responsibility.

2.3 Missing Type Safety on Blockchain Data

XRPL API responses were typed as any throughout the codebase — in edge functions, in React hooks, and in components. Field name bugs like tx.Amount vs tx.amount were undetectable at compile time. Currency decoding logic (decodeCurrency) was duplicated inline in multiple components, leading to inconsistent display of the same token.

Lesson: Blockchain response types must be explicitly defined as TypeScript interfaces and shared across the entire codebase via a central xrpl-types.ts library. All XRPL utility functions (decode currency, format amount, build explorer URL) must live in a single shared module.

2.4 No Multi-Tenancy From Day One

The V1 schema was designed for a single operator with no tenant_id on any table. Adding multi-tenancy to an existing schema requires migrating every table, rewriting every RLS policy, updating every edge function to extract tenant context from the JWT, and changing every query. This is a multi-week refactor with significant regression risk.

Lesson: If multi-tenancy is a goal at any point in a project's roadmap, the tenant_id column must be present in every user-facing table from the very first migration. Retrofitting tenant isolation is orders of magnitude harder than building it in from the start. V2 adds tenant_id to every table as a non-nullable foreign key before writing a single line of application code.

2.5 AI-Assisted Tooling Creates Commit Drift

The V1 codebase was generated and maintained via Lovable, an AI-assisted development platform. Over 250+ commits, the codebase accumulated patterns that contradicted each other: two tables tracking wallet identity (wallet_profiles vs user_wallets), two approaches to real-time data (WebSocket subscription AND React Query polling), hardcoded test data in production components, and polling intervals defined at different values in different files.

Lesson: AI-assisted tools accelerate early prototyping but require deliberate architectural governance to prevent drift. V2 establishes Architecture Decision Records (ADRs) for every significant pattern choice, ROSETTA.md as a living architectural reference, and automated linting rules to enforce conventions (no any in XRPL code, no direct Supabase client creation outside client.ts, etc.).

2.6 Competing Data-Fetching Patterns

V1 used both React Query polling and XRPL WebSocket subscriptions simultaneously for portfolio data. The two systems could return different snapshots of account state, causing UI flicker and race conditions. The WebSocket subscription hook (useXRPLSubscription) and React Query hook (useXRPLPortfolio) had no coordination mechanism.

Lesson: Choose one canonical data-fetching strategy per data domain and enforce it uniformly. V2 defines: React Query polling (30-second interval) for portfolio data, and XRPL WebSocket for real-time transaction confirmation feedback only (not as a portfolio data source). A single constant XRPL_POLL_INTERVAL_MS governs all polling, defined once in src/lib/constants.ts.

2.7 Sequential XRPL Requests

The xrpl-account-data edge function issued five XRPL requests sequentially with 100ms artificial delays between each. Portfolio load time was dominated by serialized round-trips: on a typical connection this meant 500ms–2s of unnecessary latency on every page load.

Lesson: Independent blockchain queries must be issued in parallel using Promise.all(). The sequential delays were added speculatively to avoid rate limiting; in practice, XRPL nodes do not rate-limit sequential requests from the same IP at this volume. If rate limiting is ever observed, implement proper exponential backoff — never fixed delays.

V2 Core Design Principles

These six principles are non-negotiable constraints that govern every design decision in V2. When two approaches are in conflict, these principles serve as the tiebreaker — in order of priority.

🔒

1. Security First

No raw private keys in any database. All signing is either user-controlled (Xaman/hardware wallet) or vault-referenced. Every privilege is least-privilege. Default deny, not default allow.

🏠

2. Multi-Tenant by Default

Every table has tenant_id. Every RLS policy enforces tenant isolation. Every API call extracts tenant context from JWT claims. No global queries ever touch cross-tenant data.

🐾

3. Type-Safe Blockchain Layer

All XRPL response shapes have explicit TypeScript interfaces. Zero any types in blockchain-facing code. Supabase types regenerated automatically on every DB migration.

⚙

4. Single Responsibility

Each module, hook, context, and edge function does exactly one thing. No god objects. Files exceeding 200 lines are a signal to refactor. Each edge function is one business action.

📊

5. Observable

Structured JSON logging on all edge functions. Rate limiting with metered counters. Immutable audit logs for all sensitive operations. Error reporting via Sentry. P95 latency monitored.

📈

6. Testable

Edge functions are pure functions with injected dependencies. Business logic is separated from transport. E2E tests cover the full tokenization pipeline. No test data in production code.

3.1 Security First — Elaborated

Security is not a feature to be added later — it is a constraint on the initial design. The V2 security model is defined by the following rules, enforced by code review and automated linting:

  • No wallet_secret column exists anywhere in the schema. If a PR introduces such a column, CI fails.
  • Signing keys never transit application code. Testnet uses a single platform faucet wallet. Mainnet uses Xaman QR (user holds their own keys). Future HSM support uses vault key IDs only.
  • JWT claims are verified on every edge function, not just via Supabase gateway. The pattern is: verify JWT → extract user_id and tenant_id → validate against DB → proceed.
  • Service role key is never exposed to the browser. The client-side Supabase client uses the anon key only. Service role is used exclusively in Deno edge functions.
  • All destructive XRPL operations require an additional confirmation step (e.g., clawback, freeze) logged to the immutable audit table before execution.

3.2 Multi-Tenant by Default — Elaborated

Tenant isolation is implemented at three independent layers, so that a failure at one layer does not compromise isolation: 

Layer 1: DATABASE (PostgreSQL RLS)
  ├── Every table has tenant_id NOT NULL REFERENCES tenants(id)
  ├── RLS enabled on every user-facing table
  └── Policy: USING (tenant_id = (SELECT tenant_id FROM profiles WHERE id = auth.uid()))

Layer 2: API (Edge Functions)
  ├── JWT token contains user_id
  ├── Edge function reads tenant_id from profiles table using service role
  ├── All queries are scoped to that tenant_id
  └── Cross-tenant queries are rejected with 403

Layer 3: FRONTEND (Next.js Middleware)
  ├── Subdomain resolved at middleware: tenant1.platform.com → tenant_slug = "tenant1"
  ├── Tenant config fetched server-side and injected as React context
  └── UI components branded per tenant; no cross-tenant navigation possible

4.3 Database Isolation Strategy

V2 uses shared schema, row-level isolation: all tenants share the same PostgreSQL database and schema, but every row in every user-facing table contains a tenant_id foreign key, and RLS policies enforce that users can only see rows belonging to their own tenant.

ℹ
Why Shared Schema (not separate databases)?

Separate databases per tenant provide the strongest isolation but cost $0–$25/month per tenant at Supabase's pricing, making small tenants economically unviable. Shared schema with RLS is the industry-standard approach for B2B SaaS on PostgreSQL. The tradeoff: a misconfigured RLS policy could expose cross-tenant data (mitigated by automated policy testing in CI).

The canonical RLS pattern used across all user-facing tables:

RLS Policy Pattern — All User-Facing Tables 
-- Enable RLS
ALTER TABLE <table> ENABLE ROW LEVEL SECURITY;

-- SELECT: user can only see rows in their tenant
CREATE POLICY "tenant_select" ON <table>
  FOR SELECT USING (
    tenant_id = (
      SELECT tenant_id FROM profiles
      WHERE id = auth.uid()
    )
  );

-- INSERT: user can only insert into their tenant
CREATE POLICY "tenant_insert" ON <table>
  FOR INSERT WITH CHECK (
    tenant_id = (
      SELECT tenant_id FROM profiles
      WHERE id = auth.uid()
    )
  );

export function middleware(req: NextRequest) {

5.5 Token Metadata Standard (XLS-89)

For larger documents (legal deeds, valuation reports, imagery), V2 uses an on-chain + off-chain hybrid: the on-chain metadata contains only a compact summary and an IPFS content-addressable hash pointing to the full document set.

5.6 Compliance Flags

MPT compliance behavior is controlled by flags set at issuance creation time. These flags are immutable after the issuance is created — plan them carefully.

If a tenant launches without CanClawback and later faces a regulatory request to reclaim tokens from a sanctioned user, they have no on-chain recourse. V2 default recommendation: set RequireAuth + CanClawback + CanLock for all regulated real estate offerings. Only omit flags for fully public, unrestricted tokens (rare in real estate context).

API / Backend Design — Edge Functions V2

All server-side business logic runs in Deno edge functions deployed on Supabase. Edge functions are the only code that can communicate with XRPL nodes, access the service role key, or reference vault key IDs. The browser never calls XRPL directly.

7.1 Universal Edge Function Pattern

Every edge function in V2 follows the same structure. Deviation from this pattern is a code review failure criterion. 

// Universal V2 edge function template
import { createClient } from "https://esm.sh/@supabase/supabase-js@2";
import { corsHeaders } from "../_shared/cors.ts";
import { log } from "../_shared/logger.ts";
import { checkRateLimit } from "../_shared/rate-limit.ts";
import { extractTenantId } from "../_shared/auth.ts";

Deno.serve(async (req: Request) => {
  // 1. CORS preflight
  if (req.method === "OPTIONS") {
    return new Response(null, { headers: corsHeaders });
  }

  const requestId = crypto.randomUUID();
  const startTime = Date.now();

  try {
    // 2. Authenticate: verify JWT, extract user
    const authHeader = req.headers.get("Authorization");
    if (!authHeader) return new Response("Unauthorized", { status: 401 });

    const supabaseUser = createClient(
      Deno.env.get("SUPABASE_URL")!,
      Deno.env.get("SUPABASE_ANON_KEY")!,
      { global: { headers: { Authorization: authHeader } } }
    );
    const { data: { user }, error: authError } = await supabaseUser.auth.getUser();
    if (authError || !user) return new Response("Unauthorized", { status: 401 });

    // 3. Extract tenant_id from user profile (service role)
    const supabaseAdmin = createClient(
      Deno.env.get("SUPABASE_URL")!,
      Deno.env.get("SUPABASE_SERVICE_ROLE_KEY")!
    );
    const tenantId = await extractTenantId(supabaseAdmin, user.id);
    if (!tenantId) return new Response("Tenant not found", { status: 403 });

    // 4. Rate limit check
    const limited = await checkRateLimit(supabaseAdmin, user.id, "function-name");
    if (limited) return new Response("Too Many Requests", { status: 429 });

    // 5. Business logic (function-specific)
    const body = await req.json();
    const result = await businessLogic(supabaseAdmin, user, tenantId, body);

    // 6. Structured log
    log.info("function-name.success", {
      requestId, userId: user.id, tenantId,
      durationMs: Date.now() - startTime
    });

    return new Response(JSON.stringify(result), {
      headers: { ...corsHeaders, "Content-Type": "application/json" }
    });

  } catch (error) {
    log.error("function-name.error", {
      requestId, error: error.message,
      durationMs: Date.now() - startTime
    });
    return new Response(
      JSON.stringify({ error: "Internal Server Error" }),
      { status: 500, headers: { ...corsHeaders, "Content-Type": "application/json" } }
    );
  }
});

Wallet & Key Management — Security Model

⚠
The Most Critical Section of This Document

Every security decision described in this section is a response to a real issue found in V1. These rules are absolute — they are not negotiable for performance or convenience. Any PR that violates these rules must be rejected regardless of other merit.

8.1 The Golden Rule: No Raw Keys in the Database

The V1 codebase stored raw XRPL private keys (wallet_secret) in the user_wallets table. This is a critical security vulnerability: a single SQL injection, a misconfigured RLS policy, or a compromised service role key would expose every user's private keys.

In V2, the rule is absolute: no private key material of any kind ever touches a relational database column. This includes encrypted keys — if the encryption key is also stored in the DB, you have not improved the security posture. Key material must be stored in a dedicated secrets management service that:

  • Stores keys in hardware-backed encrypted storage (ideally HSM)
  • Provides key operations (sign, encrypt, decrypt) via API, never returning raw key material
  • Maintains an access log of all key usage
  • Supports key rotation without application downtime

Phase 5+ will add support for Ledger hardware wallets via WebUSB/WebHID API. In this model:

  • The @ledgerhq/hw-transport-webusb library communicates with the device from the browser

8.7 Inactivity Timeout

Wallet sessions auto-disconnect after a configurable inactivity period. In V2, this is implemented as a standalone useInactivity hook — not embedded inside the wallet context (as it was in V1). 

// useInactivity.ts — standalone hook, no responsibilities beyond inactivity tracking
export function useInactivity({
  timeoutMinutes,
  onTimeout,
}: {
  timeoutMinutes: number;
  onTimeout: () => void;
}) {
  const lastActivityRef = useRef(Date.now());
  const timerRef = useRef<ReturnType<typeof setTimeout>>();

  const resetTimer = useCallback(() => {
    lastActivityRef.current = Date.now();
    clearTimeout(timerRef.current);
    timerRef.current = setTimeout(onTimeout, timeoutMinutes * 60 * 1000);
  }, [timeoutMinutes, onTimeout]);

  useEffect(() => {
    const events = ['mousemove', 'keydown', 'click', 'touchstart'];
    events.forEach(e => window.addEventListener(e, resetTimer, { passive: true }));
    resetTimer(); // start timer on mount

    return () => {
      clearTimeout(timerRef.current);
      events.forEach(e => window.removeEventListener(e, resetTimer));
    };
  }, [resetTimer]);
}

10.3 Node Failover Strategy

XRPL nodes can be temporarily unavailable. V2 implements automatic failover with exponential backoff: 

// xrpl-client.ts (edge function shared utility)
const XRPL_NODES = {
  mainnet: [
    "wss://xrplcluster.com",
    "wss://s1.ripple.com",
    "wss://s2.ripple.com",
  ],
  testnet: [
    "wss://s.altnet.rippletest.net:51233",
    "wss://testnet.xrpl-labs.com",
  ],
};

async function withXRPLFailover<T>(
  network: 'mainnet' | 'testnet',
  operation: (client: XRPLClient) => Promise<T>
): Promise<T> {
  const nodes = XRPL_NODES[network];

  for (let attempt = 0; attempt < nodes.length; attempt++) {
    try {
      const client = new XRPLClient(nodes[attempt]);
      await client.connect();
      try {
        return await operation(client);
      } finally {
        await client.disconnect();
      }
    } catch (error) {
      if (attempt === nodes.length - 1) throw error; // exhausted all nodes
      // Exponential backoff: 100ms, 200ms, 400ms...
      const delay = 100 * Math.pow(2, attempt);
      await new Promise(resolve => setTimeout(resolve, delay));
      log.warn("xrpl.node_failover", { node: nodes[attempt], attempt, delay });
    }
  }
  throw new Error("All XRPL nodes exhausted");
}

11.3 PII Data Handling

Personal Identifiable Information (PII) collected during KYC — passport scans, selfies, address documents — is handled by the KYC provider and never stored in the platform database. The kyc_verifications table stores only:

  • KYC status (pending/approved/rejected) — no PII
  • Provider inquiry ID — opaque reference in provider's system
  • Verification date — when approval occurred
  • Data ref — optional pointer to provider's data access API (not raw PII)

KYC providers like Persona and Onfido are GDPR-compliant and handle PII retention, deletion requests, and data residency requirements on their end. This keeps the platform's data compliance burden minimal.

⚠
GDPR Right to Erasure Consideration

If a user exercises their right to erasure, the platform can delete their profile, wallet records, and KYC verification records. However, blockchain transaction history is immutable — wallet addresses and transaction amounts remain on the XRPL ledger forever. Users must be informed of this in the Terms of Service before they connect a wallet.
